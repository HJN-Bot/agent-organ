# 2026-02-09｜Hackathon 讨论记录：Agent Organ Hub（OpenClaw 硬件拓展）

> 结论先行：主亮点选 **A：输入/输出更“器官化”更立体**（多源输入 + 声/光/屏输出 + L1/L2/L3 自主分层），**B：agent-to-agent 对齐**作为工作场景加分项（总控/质控回放）。

---

## 1) 我们要解决的核心问题

- 把 OpenClaw 从“手机里的工具”升级为“外置器官/空间中枢”：
  - **输入更低摩擦、更连续**：语音/照片/文件/事件不再必须先过人脑再手动转述
  - **输出更立体、更有陪伴感**：不仅在 chat 里，而是声/光/屏的空间化反馈
- 目标不是做一个炫技硬件，而是让“事件不丢 + 可回放 + 可执行闭环”真正跑起来。

---

## 2) 自主等级（避免“越过人代替人”的恐惧）

- **L1 自动记录（默认）**：捕获→结构化→入库（不执行/不外发）
- **L2 建议输出（默认）**：给下一步建议/草稿，用户一键确认
- **L3 白名单自动执行（可选）**：仅低风险动作自动做（建提醒/建日程/写入表格等），且可撤销、可审计

> 好处：边界清晰，能解释、能治理、能扩展。

---

## 3) 方案 A（主线）：输入/输出更立体（器官感）

### 3.1 输入端扩展（Sources）
按落地难度从低到高：
- 语音（按键触发 10s / push-to-record）
- 图像（拍照快照 + OCR）
- 文件（手机分享/上传 PDF/DOCX）
- 数字事件（通知/日历/位置，作为加分项）

统一成事件 schema（核心）：
- `event_id, ts, source_type, payload_uri, text(ASR/OCR), metadata, sensitivity, trace_id`

### 3.2 输出端延伸（Sinks）
最能出 demo 效果的三件套：
- **声**：TTS 播报“一句话总结 + 下一步”
- **光**：状态灯/灯带（成功/待确认/风险）
- **屏**：小屏/大屏 dashboard（空间化展示）

---

## 4) 技术路线：先 C（手机模拟穿戴）→ 再 A（BLE 穿戴）

### 4.1 为什么先 C
- 48h 内真正决定成败的是：采集→事件→处理→反馈的闭环、延迟指标、失败模式与交互设计。
- 手机天然具备：摄像头/麦克风/网络/TTS/权限/通知体系 → 可快速验证 80% 体验与指标。

### 4.2 A（BLE 穿戴）应当“瘦身”
- BLE 只传小消息（按钮/状态/电量/振动命令），**不要走 BLE 传音频/图片**。
- 重计算仍在手机/边缘盒子：穿戴端做“触发器 + 反馈器 + 存在感”。

### 4.3 48h 可跑通的架构（C 阶段）
- Phone（采集 + ASR/OCR + 事件化）→ OpenClaw（分类/入库/自动化）→ 输出（TTS/灯/屏）
- 全链路打点：t_capture / t_infer / t_send / t_recv / t_notify，输出 P50/P90 延迟

---

## 5) 隐私/合规：MVP Privacy by Design（必须先讲清楚）

会被质疑：偷录偷拍、数据去哪、第三人隐私/商业机密、被滥用为监控。

最小可行方案（可验证的约束）：
- 默认不连续采集；**按键采集（push-to-record/capture）**
- **不可关闭的外部指示灯**（录音/录像/上传分状态）+ 声/振提示
- 本地优先：默认只上传事件元数据；原始音视频需二次确认
- 可撤回窗口（误触友好）
- 审计与回放：采集开始/结束、触发方式、是否上传、访问者留痕
- 数据保护：加密、最短留存、一键清除/远程擦除

工作场景红线：
1) 不做隐形监控；2) 会议/客户现场默认禁用，除非明确同意；3) 访问必须审批+审计。

---

## 6) 工作场景加分项（B 的最小版本）：总控/质控回放

不做复杂“两个 agent 自己聊天”，只做“多事件汇总对齐”：
- 每个人的个人代理（PA）捕获 event
- 团队对齐代理（TAA）汇总：一致点/分歧点/待确认点/行动项

---

## 7) MVP 验证指标（先手机端 C）

北极星：**每周闭环完成数**（由 Hub 触发/辅助的事项，最终被完成/执行成功）。

漏斗指标：
- 捕获数（可执行意图）→ 结构化转化率 → 执行成功率 → 跟进完成率 → 复盘确认率

体验指标（防卸载）：
- 提醒命中率（看到后是否采取动作）
- 确认成本（步数/秒数）
- 负反馈率（烦/不相关/关闭提醒）
- 7日/28日留存（高权限用户尤重要）

立项 BLE（A）前门槛：
- 证明“确认成本”是主要瓶颈（大量流失发生在解锁/打开App）
- 证明“打断时机”影响命中率（驾驶/会议/运动等状态下表现差）

---

## 8) 对外质疑点 & 反驳要点（可用于路演）

- “手机够了，为什么硬件？”→ 先手机闭环验证；硬件用于把确认降到 1–3 秒并提升打断决策
- “会不会变提醒轰炸机？”→ 提醒预算 + 升级策略（少提醒、但更准）
- “LLM 不可靠怎么办？”→ 高风险强确认；低风险白名单自动化；可撤销/回滚；每次动作有因果链

---

## 9) 下一步（建议）

1) 先定 C 阶段：优先语音还是视觉？优先 iOS 还是 Android？（以及是否接受前台演示）
2) 用 48h 里程碑推进：每 6h 出一个可演示增量
3) 把“隐私默认策略+话术”写进 demo（反而加分）
4) B（团队对齐）只做汇总回放，不做复杂自治
